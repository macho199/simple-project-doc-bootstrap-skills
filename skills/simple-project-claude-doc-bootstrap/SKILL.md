# simple-project-claude-doc-bootstrap

## 목적

이 스킬은 장기적인 AI 개발 작업에서 프로젝트 문서를 최소한으로 유지하면서도,
AI가 프로젝트의 목표, 현재 상태, 구조, 작업 규칙을 잃지 않고 이어서 작업할 수 있게 돕는다.

이 스킬의 핵심은 문서를 많이 만드는 것이 아니라,
꼭 필요한 문서만 만들고 지속적으로 최신 상태로 유지하는 것이다.

문서는 과거 기록을 길게 보존하는 저장소가 아니라,
현재 프로젝트를 올바르게 이해하고 다음 작업을 이어가기 위한 기준 문서로 유지한다.

---

## 언제 사용할까

다음과 같은 상황에서 이 스킬을 사용한다.

- 새 프로젝트를 시작할 때
- AI가 여러 세션에 걸쳐 같은 프로젝트를 계속 작업해야 할 때
- 프로젝트 문서가 없어서 시작점이 모호할 때
- 문서가 너무 흩어져 있어서 최소 구조로 정리하고 싶을 때
- AI가 현재 상태를 자주 놓치는 프로젝트를 안정화하고 싶을 때
- 기존 문서가 많지만 무엇이 기준 문서인지 불분명할 때

---

## 핵심 원칙

- 문서는 적을수록 좋다.
- 새 문서를 만들기 전에 기존 문서에 통합 가능한지 먼저 검토한다.
- 문서는 과거 이력을 길게 누적하는 용도가 아니라 현재 상태를 반영하는 용도로 유지한다.
- 오래된 내용은 남겨두지 말고 갱신하거나 제거한다.
- 사용자의 최신 요구사항이 기존 문서보다 우선한다.
- CLAUDE.md에는 모든 세션에 공통 적용되는 규칙만 유지한다.
- 의미 있는 작업이 끝난 뒤에는 관련 문서를 반드시 현행화한다.
- 문서의 양보다 정확성, 최신성, 역할 분리를 우선한다.

---

## 기본 관리 대상 문서

이 스킬은 기본적으로 아래 4개의 문서만 관리한다.

- `CLAUDE.md`
- `PROJECT.md`
- `STATUS.md`
- `ARCHITECTURE.md`

사용자가 명시적으로 요청하지 않는 한, 추가 문서를 만들지 않는다.

---

## 문서별 역할

### CLAUDE.md
프로젝트 전반에 공통으로 적용되는 작업 규칙을 기록한다.

포함할 수 있는 내용:
- 작업 방식
- 수정 원칙
- 테스트/검증 습관
- 문서 관리 규칙
- AI가 먼저 읽어야 할 파일

포함하지 말아야 할 내용:
- 특정 작업에만 적용되는 지시사항
- 임시 계획
- 상세 TODO
- 자주 바뀌는 진행 상황
- 긴 배경 설명
- 기능 단위 구현 메모

### PROJECT.md
프로젝트의 목표, 범위, 제약사항, 성공 기준을 기록한다.

포함 예시:
- 프로젝트 목표
- 배경
- 포함 범위 / 제외 범위
- 핵심 기능
- 성공 기준
- 제약사항

### STATUS.md
현재 시점 기준의 진행 상황을 요약한다.

포함 예시:
- 현재 상태 요약
- 완료된 일
- 진행 중인 일
- 다음 작업
- 막힌 점
- 최근 중요한 변경

주의:
- 긴 작업 로그처럼 누적하지 않는다.
- 현재 상태 중심으로 재작성한다.
- 과거보다 지금 무엇이 사실인지가 더 중요하다.

### ARCHITECTURE.md
현재 확정된 구조와 주요 설계 결정을 기록한다.

포함 예시:
- 시스템 개요
- 주요 모듈
- 디렉토리 구조
- 핵심 설계 결정
- 기술 선택 이유

주의:
- 검토 중인 아이디어보다 현재 채택된 구조를 우선 기록한다.
- 폐기된 설계안은 오래 남겨두지 않는다.
- 현재 코드와 실제 구조를 기준으로 작성한다.

## 실행 절차

이 스킬이 호출되면 다음 순서로 진행한다.

### 1. 현재 상태 파악
- 저장소에 기존 문서가 있는지 확인한다.
- 기존 문서가 있으면 중복 생성하지 않는다.
- 기존 문서의 역할이 겹치거나 내용이 오래되었는지 확인한다.
- 현재 코드와 저장소 구조를 가능한 범위에서 함께 확인한다.

### 2. 최소 문서 세트 초기화
필요할 경우 아래 파일만 생성한다.

- `CLAUDE.md`
- `PROJECT.md`
- `STATUS.md`
- `ARCHITECTURE.md`

문서 내용은 `templates/` 아래 템플릿을 기반으로 작성한다.

### 3. 현재 프로젝트에 맞게 초안 작성
사용자의 요구사항, 저장소 구조, 현재 코드 상태를 바탕으로
각 문서의 초안을 구체화한다.

확인되지 않은 내용은 사실처럼 단정하지 않는다.
추정이 필요한 경우에는 열린 항목 또는 확인 필요 항목으로 남긴다.

### 4. 작업 이후 문서 현행화
의미 있는 작업이 끝날 때마다 다음 기준으로 문서를 갱신한다.

- 작업 규칙 변경: `CLAUDE.md`
- 목표/범위 변경: `PROJECT.md`
- 진행 상황 변경: `STATUS.md`
- 구조/설계 변경: `ARCHITECTURE.md`

문서 현행화는 선택 사항이 아니라 작업 완료의 일부로 간주한다.

### 5. 문서 증식 방지
새 문서를 만들기 전에 반드시 아래를 검토한다.

- 이 내용을 기존 문서의 섹션으로 넣을 수 있는가?
- 이 문서가 앞으로 여러 세션 동안 계속 중요할 것인가?
- 새 파일이 생기면 AI의 문맥 부담이 더 커지지 않는가?

위 질문에 명확하게 “예”라고 답할 수 없으면 새 문서를 만들지 않는다.

## CLAUDE.md 작성 규칙

- CLAUDE.md에는 정해진 형식은 없지만, 짧고 간결하며 사람이 읽기 쉽게 유지한다.
- CLAUDE.md는 모든 세션에서 로드되므로, 프로젝트 전반에 공통으로 적용되는 규칙만 포함한다.
- 작업별 지시사항, 임시 계획, 진행 중인 세부 내용은 CLAUDE.md에 넣지 않는다.
- 프로젝트 목표는 `PROJECT.md`, 현재 진행상황은 `STATUS.md`, 구조와 설계는 `ARCHITECTURE.md`에 기록한다.

---

## 공통 문서 작성 규칙

- 중복 내용을 여러 문서에 반복하지 않는다.
- 긴 서술보다 구조화된 헤더와 짧은 문장을 선호한다.
- 오래된 정보는 남겨두기보다 현재 기준으로 갱신한다.
- 문서는 사람이 읽어도 이해되도록 유지한다.
- 문서의 길이보다 정확성과 최신성을 우선한다.
- 확인되지 않은 내용을 사실처럼 쓰지 않는다.
- 문서 형식은 복잡함보다 유지 가능성을 우선한다.

---

## 사용자 요구사항 우선 규칙

- 사용자의 최신 확정 요구사항을 프로젝트의 최우선 기준으로 삼는다.
- 기존 문서와 사용자의 최신 지시가 충돌하면 문서를 수정하여 최신 지시를 반영한다.
- 예전에 기록된 계획이 있더라도, 현재 사용자 요구와 다르면 그대로 따르지 않는다.
- 문서는 사용자 요구사항을 보조하는 수단이지, 사용자 지시를 덮어쓰는 기준이 아니다.

---

## 안티패턴 / 금지 행동

- CLAUDE.md를 프로젝트 설명서, 작업 로그, TODO 모음처럼 사용하지 않는다.
- STATUS.md를 날짜별 작업 일지처럼 끝없이 누적하지 않는다.
- ARCHITECTURE.md에 폐기된 설계안이나 검토만 했던 아이디어를 길게 남겨두지 않는다.
- 같은 내용을 여러 문서에 중복 기록하지 않는다.
- 기존 4개 문서에 담을 수 있는 내용을 위해 새 문서를 만들지 않는다.
- 사용자의 최신 요구사항보다 오래된 문서를 우선하지 않는다.
- 확인되지 않은 내용을 사실처럼 문서화하지 않는다.
- 문서 형식을 과도하게 복잡하게 만들지 않는다.
- 의미 있는 변경 후 문서 갱신을 미루지 않는다.
- 현재 작업에 도움이 되지 않는 오래된 내용을 불필요하게 보존하지 않는다.

---

## 기대 결과

이 스킬 실행 후에는 다음 상태를 목표로 한다.

- AI가 다음 세션에서도 프로젝트를 빠르게 이해할 수 있다.
- 어떤 문서가 무엇의 기준 문서인지 명확하다.
- 프로젝트 목표, 현재 상태, 구조, 작업 규칙이 분리되어 있다.
- 문서 수가 과도하게 늘어나지 않는다.
- 오래된 계획보다 현재 기준의 문서가 유지된다.
- 코드와 문서의 불일치가 줄어든다.

---

## 금지 사항

- 같은 내용을 여러 문서에 중복 기록하지 않는다.
- STATUS.md를 긴 일지처럼 사용하지 않는다.
- CLAUDE.md에 프로젝트 상세 배경이나 작업별 TODO를 넣지 않는다.
- ARCHITECTURE.md에 폐기된 설계안을 길게 남겨두지 않는다.
- 사용자가 요청하지 않은 문서 세트를 자동으로 늘리지 않는다.
- 추정과 사실을 구분 없이 기록하지 않는다.

---

## 템플릿 사용 규칙

문서를 새로 생성하거나 크게 재작성할 때는 `templates/` 아래 템플릿을 출발점으로 사용한다.

- `templates/CLAUDE.template.md`
- `templates/PROJECT.template.md`
- `templates/STATUS.template.md`
- `templates/ARCHITECTURE.template.md`

단, 템플릿을 기계적으로 채우지 말고 실제 프로젝트 상태에 맞게 간결하게 조정한다.

---

## 출력 품질 기준

이 스킬이 생성하거나 수정하는 문서는 아래 기준을 만족해야 한다.

- 서로 역할이 겹치지 않아야 한다.
- 사람이 빠르게 읽고 이해할 수 있어야 한다.
- 다음 세션의 AI가 곧바로 이어서 작업할 수 있어야 한다.
- 현재 프로젝트 상태를 기준으로 유지되어야 한다.
- 불필요하게 길거나 장황하지 않아야 한다.

---

## 실행 시 기본 판단 기준

문서화 중 판단이 필요하면 아래 우선순서를 따른다.

1. 사용자의 최신 요구사항
2. 현재 코드와 저장소 구조
3. 기존 문서 중 최신성과 역할이 명확한 문서
4. 템플릿 기본 구조

기존 문서가 많고 서로 충돌할 경우,
더 최신이며 현재 코드와 일치하는 내용을 우선한다.
오래되었거나 중복되는 내용은 통합, 축약, 갱신 또는 제거한다.
