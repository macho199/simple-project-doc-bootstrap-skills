# simple-project-doc-bootstrap

## 목적

이 스킬은 장기적인 AI 개발 작업에서 프로젝트 문서를 최소한으로 유지하면서도,
AI가 프로젝트의 목표, 현재 상태, 구조, 작업 규칙을 잃지 않고 이어서 작업할 수 있게 돕는다.

핵심은 문서를 많이 만드는 것이 아니라,
꼭 필요한 문서만 만들고 지속적으로 최신 상태로 유지하는 것이다.

## 언제 사용할까

- 새 프로젝트를 시작할 때
- AI가 여러 세션에 걸쳐 같은 프로젝트를 계속 작업해야 할 때
- 프로젝트 문서가 없어서 시작점이 모호할 때
- 문서가 너무 흩어져 있어서 최소 구조로 정리하고 싶을 때
- 기존 문서가 많지만 무엇이 기준 문서인지 불분명할 때

## 핵심 원칙

- 문서는 적을수록 좋다.
- 새 문서를 만들기 전에 기존 문서에 통합 가능한지 먼저 검토한다.
- 문서는 과거 이력을 길게 누적하는 용도가 아니라 현재 상태를 반영하는 용도로 유지한다.
- 오래된 내용은 남겨두지 말고 갱신하거나 제거한다.
- 사용자의 최신 요구사항이 기존 문서보다 우선한다.
- 의미 있는 작업이 끝난 뒤에는 관련 문서를 반드시 현행화한다.

## 대상 에이전트 선택 규칙

이 스킬은 시작할 때 먼저 이 프로젝트가 어느 에이전트 기준으로 운영될지 확정한다.

- 사용자가 이미 `Codex` 또는 `Claude Code`를 명시했다면 그대로 따른다.
- 명시되지 않았다면 짧게 한 번 묻는다.
  이 프로젝트 문서는 `Codex` 기준으로 운영할지, `Claude Code` 기준으로 운영할지 확인한다.
- `Codex`를 선택하면 규칙 문서는 `AGENTS.md`다.
- `Claude Code`를 선택하면 규칙 문서는 `CLAUDE.md`다.
- 사용자가 명시적으로 요청하지 않는 한 `AGENTS.md`와 `CLAUDE.md`를 동시에 새로 만들지 않는다.

## 기본 관리 대상 문서

항상 아래 3개 문서는 공통으로 관리한다.

- `PROJECT.md`
- `STATUS.md`
- `ARCHITECTURE.md`

그리고 대상 에이전트에 따라 아래 둘 중 하나만 관리한다.

- Codex: `AGENTS.md`
- Claude Code: `CLAUDE.md`

프로젝트에 결과물(report, 데이터 파일, API 응답 등)이 정의된 경우에는 아래도 함께 관리한다.

- `fixtures/` — 결과물의 형식과 구조를 보여주는 최소 예시 데이터 파일 모음

## 문서별 역할

### 규칙 문서
선택된 에이전트 기준의 공통 작업 규칙을 기록한다.

- Codex면 `AGENTS.md`
- Claude Code면 `CLAUDE.md`

포함할 수 있는 내용:
- 작업 방식
- 수정 원칙
- 테스트/검증 습관
- 문서 관리 규칙
- AI가 먼저 읽어야 할 파일
- 결과물이 있는 프로젝트의 경우, `fixtures/` 폴더를 참조하도록 안내
  - `fixtures/`에는 결과물의 형식과 구조를 보여주는 최소 예시 데이터 파일을 둔다.
  - AI가 세션마다 결과물 형식을 다르게 해석하지 않도록 기준점을 고정하는 역할을 한다.

포함하지 말아야 할 내용:
- 특정 작업에만 적용되는 지시사항
- 임시 계획
- 상세 TODO
- 자주 바뀌는 진행 상황
- 긴 배경 설명
- 기능 단위 구현 메모

### PROJECT.md
프로젝트의 목표, 범위, 제약사항, 성공 기준을 기록한다.

### STATUS.md
현재 시점 기준의 진행 상황을 요약한다.

- 긴 작업 로그처럼 누적하지 않는다.
- 현재 상태 중심으로 재작성한다.

### ARCHITECTURE.md
현재 확정된 구조와 주요 설계 결정을 기록한다.

- 현재 코드와 실제 구조를 기준으로 작성한다.
- 폐기된 설계안은 오래 남겨두지 않는다.

## 실행 절차

### 1. 현재 상태 파악
- 저장소에 기존 문서가 있는지 확인한다.
- 현재 코드와 저장소 구조를 가능한 범위에서 함께 확인한다.
- 기존 문서의 역할이 겹치거나 내용이 오래되었는지 확인한다.

### 2. 대상 에이전트 확정
- 사용자 요청에 `Codex` 또는 `Claude Code`가 이미 있으면 그대로 사용한다.
- 없으면 먼저 어떤 워크플로우를 쓸지 묻는다.
- 프로젝트에 이미 `AGENTS.md` 또는 `CLAUDE.md`가 있고 방향이 명확하면 이를 참고한다.
- 둘 다 있고 내용이 충돌하거나, 기존 문서와 사용자의 새 방향이 다르면 큰 수정 전에 확인한다.

### 3. 최소 문서 세트 초기화
필요할 경우 아래 파일만 생성한다.

- Codex 선택 시: `AGENTS.md`, `PROJECT.md`, `STATUS.md`, `ARCHITECTURE.md`
- Claude Code 선택 시: `CLAUDE.md`, `PROJECT.md`, `STATUS.md`, `ARCHITECTURE.md`

문서 내용은 `templates/` 아래 템플릿을 기반으로 작성한다.

프로젝트에 결과물(report, 데이터 파일, API 응답 등)이 정의된 경우,
`fixtures/` 폴더를 함께 생성하고 결과물의 최소 예시 데이터 파일을 추가한다.

### 4. 현재 프로젝트에 맞게 초안 작성
- 사용자의 요구사항, 저장소 구조, 현재 코드 상태를 바탕으로 각 문서를 구체화한다.
- 확인되지 않은 내용은 사실처럼 단정하지 않는다.
- 추정이 필요한 경우에는 열린 항목 또는 확인 필요 항목으로 남긴다.
- 프로젝트에 결과물(report, 데이터 파일, API 응답, 화면 출력 등)이 정의되어 있다면,
  `fixtures/` 폴더에 결과물의 형식과 구조를 보여주는 최소 예시 데이터 파일을 생성한다.
  규칙 문서에는 `fixtures/`를 참조하도록 안내하여,
  AI가 매 세션마다 결과물 형식을 새로 추정하지 않도록 한다.

### 5. 작업 이후 문서 현행화
- 규칙 변경: 선택된 규칙 문서
- 목표/범위 변경: `PROJECT.md`
- 진행 상황 변경: `STATUS.md`
- 구조/설계 변경: `ARCHITECTURE.md`
- 결과물 형식 변경: `fixtures/` 아래 해당 예시 파일

문서 현행화는 선택 사항이 아니라 작업 완료의 일부로 간주한다.

### 6. 문서 증식 방지
새 문서를 만들기 전에 반드시 아래를 검토한다.

- 이 내용을 기존 문서의 섹션으로 넣을 수 있는가?
- 이 문서가 앞으로 여러 세션 동안 계속 중요할 것인가?
- 새 파일이 생기면 AI의 문맥 부담이 더 커지지 않는가?

위 질문에 명확하게 “예”라고 답할 수 없으면 새 문서를 만들지 않는다.

## 템플릿 사용 규칙

문서를 새로 생성하거나 크게 재작성할 때는 `templates/` 아래 템플릿을 출발점으로 사용한다.

- 규칙 문서 공통 템플릿: `templates/RULE_DOC.template.md`
- 공통: `templates/PROJECT.template.md`
- 공통: `templates/STATUS.template.md`
- 공통: `templates/ARCHITECTURE.template.md`

규칙 문서 템플릿을 사용할 때는 선택된 대상 에이전트에 맞게
`{RULE_DOC_NAME}`을 `AGENTS.md` 또는 `CLAUDE.md`로 치환한 뒤 사용한다.

템플릿을 기계적으로 채우지 말고 실제 프로젝트 상태에 맞게 간결하게 조정한다.

## 안티패턴 / 금지 행동

- 규칙 문서를 프로젝트 설명서, 작업 로그, TODO 모음처럼 사용하지 않는다.
- `STATUS.md`를 날짜별 작업 일지처럼 끝없이 누적하지 않는다.
- `ARCHITECTURE.md`에 폐기된 설계안이나 검토만 했던 아이디어를 길게 남겨두지 않는다.
- 같은 내용을 여러 문서에 중복 기록하지 않는다.
- 기존 문서에 담을 수 있는 내용을 위해 새 문서를 만들지 않는다.
- 사용자의 최신 요구사항보다 오래된 문서를 우선하지 않는다.
- 확인되지 않은 내용을 사실처럼 문서화하지 않는다.

## 기대 결과

- AI가 다음 세션에서도 프로젝트를 빠르게 이해할 수 있다.
- 어떤 문서가 무엇의 기준 문서인지 명확하다.
- 프로젝트 목표, 현재 상태, 구조, 작업 규칙이 분리되어 있다.
- 문서 수가 과도하게 늘어나지 않는다.
- 오래된 계획보다 현재 기준의 문서가 유지된다.
